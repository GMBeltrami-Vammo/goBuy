# MCP Token Optimizer

This skill replaces expensive repeated MCP calls with lightweight Python scripts that fetch data
directly from the underlying APIs and cache results locally as JSON files.

## Why this matters

Every MCP tool call that returns a large response (table schemas, query results, file content, page
trees) consumes tokens **twice**: once for the raw response Claude must parse, and again for the
context it carries forward. A Python script that returns only the 3-5 values actually needed can
cut a 1,500-token MCP response down to a 40-token file read.

A cached result is even cheaper: if the data doesn't change between turns, Claude reads a local
`.json` file from `.mcp_cache/` — essentially free.

If the user is in a hurry, default to: optimize **all detected MCPs**, infer common queries from
the conversation history, use the current directory, and read credentials from env vars.

### Step 3: Generate `mcp_local_tools.py`

Create a single file at `<project_root>/mcp_local_tools.py` with this structure:

```python
#!/usr/bin/env python3
"""
MCP Local Tools — generated by mcp-token-optimizer skill
Replace expensive MCP calls with direct API calls + local JSON cache.

Usage:
  python mcp_local_tools.py <command> [args...]

Commands are listed per-integration below.
"""

import os, json, time, hashlib, argparse
from pathlib import Path
from datetime import datetime

CACHE_DIR = Path(".mcp_cache")
CACHE_DIR.mkdir(exist_ok=True)

def _cache_get(key: str, ttl: int) -> dict | None:
    path = CACHE_DIR / f"{key}.json"
    if path.exists():
        data = json.loads(path.read_text())
        if time.time() - data.get("_cached_at", 0) < ttl:
            return data
    return None

def _cache_set(key: str, data: dict) -> dict:
    data["_cached_at"] = time.time()
    data["_cached_ts"] = datetime.utcnow().isoformat()
    (CACHE_DIR / f"{key}.json").write_text(json.dumps(data, default=str, indent=2))
    return data

# ── [MCP integrations generated below] ──────────────────────────────────────
```

After the base cache utilities, add one section per integration. See the reference templates in
`references/` for per-integration code patterns. Choose the right template based on which MCPs
are active.

### Step 4: Build the per-integration section

For each MCP, generate functions following the pattern in the corresponding reference file:
- `references/supabase_template.py` → Supabase (execute_sql, storage reads)
- `references/metabase_template.py` → Metabase (run saved cards, MBQL queries)
- `references/notion_template.py` → Notion (page content, database rows)
- `references/drive_template.py` → Google Drive (file content, sheet data)

Each integration gets a clearly named function per operation:
- Good: `get_active_subscriptions()`, `get_churn_rate_monthly()`
- Bad: `run_query_1()`, `fetch_data()`

Name functions after the **business concept**, not the MCP tool. Infer these names from the
conversation history.

## Token Estimation Guidelines

Use these rough estimates when calculating savings (adjust up/down based on observed response sizes):

| MCP Tool | Typical Raw Response | Filtered Output |
|----------|---------------------|-----------------|
| Supabase execute_sql (table query) | 800–3,000 tokens | 50–150 tokens |
| Metabase query (card result) | 1,500–5,000 tokens | 40–200 tokens |
| Notion fetch (full page) | 500–2,000 tokens | 30–100 tokens |
| Google Drive read_file (doc) | 1,000–10,000 tokens | depends on content |
| Slack read_channel (history) | 500–3,000 tokens | 50–200 tokens |

The biggest wins come from **queries called 3+ times per session** and **schema/metadata lookups
that never change** (table definitions, metric card IDs, org structures).

## Caching Strategy

Guide the user on TTL choices:
- `ttl=0` — no cache, always fresh (use for write-heavy data)
- `ttl=300` — 5 min, good for operational metrics (default)
- `ttl=3600` — 1 hr, good for weekly/monthly aggregates
- `ttl=86400` — 24 hr, good for schema/metadata that rarely changes
- `ttl=float('inf')` — permanent, good for static reference data (card IDs, user lists)

## Important: Credentials Security

Always remind the user:
- Add `.mcp_cache/` to `.gitignore` (it may contain sensitive query results)
- Never commit actual credentials — only `.env.example` with placeholder values
- If using Supabase service role keys, prefer a read-only role for these scripts
- Metabase session tokens expire — recommend storing username/password and auto-refreshing
